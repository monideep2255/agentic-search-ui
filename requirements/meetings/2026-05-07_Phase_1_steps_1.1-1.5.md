## Step 1.1: strategic foundation (3 sources)

### What we reviewed

Innovation proposal, NCBI strategic alignment docs (FY26 guiding principles, Gold Standard Science, AI Action Plan), and the two-track plan (personal build vs. NCBI innovation proposal).

### Key discussion

The innovation proposal is the starting ground for V1, not aspirational. All scope elements apply, including all 5 delivery formats (web UI, GraphQL API, MCP server, KGX export, CLI agent). NCBI strategic directives (simplify discovery, reproducibility, transparency, AI-ready datasets) are hard requirements that constrain system behavior.

Django vs. FastAPI: the proposal specifies Django (NCBI standard), but the agent loop requires native async. FastAPI chosen for V1. Porting to Django for NCBI Track 2 is bounded work (HTTP layer only, ~10-15 files).

Hybrid API surface: REST+SSE for chat (conversational, streaming), GraphQL via Strawberry for programmatic access to nested biomedical data. Both served from the same FastAPI app.

Open-source LLMs preferred for the running system. Commercial models reserved for development debugging. Harness engineering philosophy: a strong harness reduces the capability required from each model.

OpenRouter as inference provider (unified API gateway, one key, one billing). LiteLLM as the in-code SDK abstraction (typed calls, cost tracking, retry logic).

Budget revised from $300-500 to ~$100 for 2-month build. Hetzner ~$60, Railway free, LangSmith free tier, LLM inference single digits/month.

### Decisions made (9)

- Innovation proposal as starting ground
- All 5 delivery formats
- FastAPI over Django
- Hybrid API surface
- NCBI directives as hard requirements
- Open-source LLMs
- OpenRouter via LiteLLM
- ~$100 budget
- Agent architecture deferred to 1.2

## Step 1.2: architecture design (6 sources)

### What we reviewed

System 3 architecture brainstorming doc, three-layer data architecture, architecture QA, CQ planning framework, NCBI databases/APIs reference, NCBI repos deep dive (13 repos analyzed).

### Key discussion

Agent architecture: single orchestrator with 3-tier model routing (guard/plan/synth). Not multi-agent. Multi-model is not multi-agent. Sub-query decomposition (LangGraph subgraph) as the upgrade path for complex queries needing 10+ tool calls. Trigger: >20% failure rate on deep research queries.

Streaming: typed SSE events (status, tool_result, token, citation, done) with hard latency budgets. Lookup 5s, single-hop 10s, multi-hop 30s, deep research 2min. If exceeded, synthesize with partial results.

Three-layer data confirmed: Layer 1 (graph, 115M nodes, <10ms), Layer 2 (NCBI APIs, 100-500ms), Layer 3 (enrichment, 200ms-2s). No changes needed.

NCBI API rate limit upgraded from 10 to 100 rps (admin access). Architecture unchanged but headroom significantly relaxed.

Two-API strategy for Layer 2: E-utilities (ClinVar, PubMed, OMIM) + Datasets API v2 (Gene, Genome, Orthologs) + Variation Services (dbSNP at 1 rps).

PubTator3 REST API replaces all local NER/normalization (GNorm2, tmVar3, AIONER). One endpoint, free, no auth, avoids 65GB+ memory.

V1 tool mapping established: cypher_query (Layer 1), ncbi_efetch (Layer 2), ncbi_dbsnp (Layer 2), pubtator_annotate (Layer 3), litvar2_lookup (Layer 3).

### Decisions made (9)

- Single orchestrator with 3-tier routing
- Model selection deferred to Phase 6
- Typed SSE events with latency budgets
- Three-layer data confirmed
- Parallel tool execution
- Result compression
- 100 rps rate limit
- Two-API strategy
- PubTator3 replaces local NER

## Step 1.3: reference implementations (2 sources)

### Sources reviewed

NCBI KG reference repo (ncbi-kg branch, 2-day prototype with NL-to-Cypher pipeline, React frontend, LangSmith tracing, 234 tests) and contractor's 8-layer NLQ architecture (typed IR, GraphQL surface, context-pack builder, NFR baseline).

### Key discussion

Monideep's position: the NCBI KG repo is a starting template for infrastructure, not an architecture to copy. The NL-to-Cypher pipeline was a quick prototype. The value is in the surrounding boilerplate: React chat components, LangSmith wiring, guardrail disclaimers, test organization, MCP server patterns.

On the contractor architecture: "way too intense." Adopt individual ideas (query classification, schema slicing, provenance-first), skip the full 8-layer system and typed IR.

Key learning from contractor: natural language should never directly generate Cypher. There must be a separation layer. Plan step reasons about what to query (structured output), tools handle how to query it.

Monideep wants to combine the reference repo's React patterns with NCBI screenshots (to be provided) for the UI design baseline.

### Decisions made (5)

- NCBI KG repo as infrastructure template
- Simplified query classification
- Schema slicing as lightweight function
- Contractor's 8-layer not adopted
- NL-to-Cypher separation principle

## Step 1.4: data handoff (2 sources)

### What we reviewed

System 1+2 data engineering repo (schema, server reference doc) and V1 shoring-up recommendations (6 fixes across 3 priority tiers).

### Key discussion

Verified data handoff: 10 node types, 14 edge predicates, 9 CURIE prefixes, connection pattern, provenance guarantees. Everything matches.

Monideep challenged the framing of "blockers." None of the 6 shoring-up items prevent System 3 from querying. All are data pipeline fixes for the next graph reload. The word "blocker" was wrong.

This led to a resilience principle: the three-layer architecture is not just a data access pattern, it's a fallback strategy. Layer 1 for speed, Layer 2 for correction, Layer 3 for enrichment.

Monideep's hard requirement: "The user must never see nothing. If something bad is happening, it just doesn't show them the message. That should never happen." A degraded answer with an explanation is always better than a blank screen.

### Decisions made (3)

- V1 shoring-up items not blockers (tag for next reload)
- Layer 2 as authoritative fallback
- Graceful degradation (never show nothing)

## Step 1.5: agent and harness research (22 sources)

### What we reviewed

All 22 research documents in the agent/harness engineering collection. Two passes: first surface-level extraction, then deep dive on harness architecture after Monideep pushed for more depth on tools, memory/SOUL, MCPs, and API patterns.

### Key discussion

Monideep's core framing: "LLMs are just a brain. I want it to be as deterministic as possible to be financially viable."

11 harness components mapped against our architecture. Three gaps identified: memory architecture (SOUL.md pattern), verification loops, prompt construction hierarchy.

The critical decision: LLM is the narrative controller only. One job: take pre-assembled cited data and explain it in natural language. The harness (deterministic code) handles everything else.

Monideep: "The LLM is the narrative controller. It is not the one who is responsible. It should have a very clean, clear, defined job. I don't want the LLM to be doing multiple things at once because I know it will hallucinate."

Balance: "When I say deterministic, I also don't want it to be so deterministic that it is giving the wrong answer all the time. There has to be balance." LLM has narrative flexibility, but citations/provenance are always code-verified.

Cost implication: determinism = financial viability. The harness absorbs complexity so the model can be cheaper.

Tool design: direct Python functions inside FastAPI for V1. MCP wraps the same functions as a separate delivery format. CLI-style contract surface: narrow, strongly typed, validated arguments.

SOUL.md behavioral directives: domain rules loaded into the prompt every session.

### Personalization vision (expanded from SOUL.md discussion)

Monideep's full vision: the system should feel like a personalized research companion, not a chatbot.

1. User logs in, provides name, profession, research focus.
2. Profile fed into every query as context. Agent knows who it's talking to.
3. System tracks query types, themes, research patterns over time.
4. Return visits: "Hi Monideep, welcome back. Want to continue your research on [topic]?"
5. Each answer leads to the next question. State continuity. Feels like a research conversation.

Three files: SOUL.md (domain rules, same for all users), USER.md (per-user profile from login), MEMORY.md (per-user learned context built over time).

Connection to hook model: personalization creates investment (IKEA effect), follow-up suggestions create variable reward, return-visit greeting creates internal trigger.

V1 scope: SOUL.md required. USER.md required if auth is in V1. MEMORY.md is V1.1.

### Decisions made (4)

- LLM as narrative controller only
- Tools as direct Python functions
- SOUL.md behavioral directives
- Verification loop with rules-based checks

## Objective review (end of session)

After completing Steps 1.1-1.5, Claude conducted an objective review of all accumulated decisions. Six concerns raised:

1. Cypher generation gap: the NL-to-Cypher separation is decided but HOW Cypher gets generated isn't. Something needs to turn structured intent into actual Cypher. Monideep agreed an LLM call inside the tool is needed, with heavy validation.

2. Five delivery formats: ambitious but Monideep noted the reference repo already has working examples of most formats, built in 2 days. The boilerplate exists.

3. Budget: clarified as ~$100/month, not total. A relative marker, not a hard cap.

4. User data storage: needs a separate database. Monideep considering Neo4j, Railway, or SQLite. Decision pending (Step 1.8).

5. Concurrent users: not an immediate concern (low user volume initially). Strategy needed but not urgent.

6. Verification loop: in-memory checks only (format validation, marker matching). No HTTP calls for URL validation on the live path.

Monideep's philosophy on feasibility: "Once we have the PRD and everything set up, it should take us at most two to three weeks to build. That is why I'm putting in so much effort up front."
