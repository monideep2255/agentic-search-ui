## Step 1.6: user psychology and product design

### Agenda

1. Step 1.6: review user psychology and product design (three product-psychology sources)
2. Resolve the fourth-source discrepancy between Plan.md and the continuation prompt
3. Decide which psychological design principles become PRD requirements

### What we reviewed

Hook model and belief design (Nir Eyal), the enterprise AI adoption gap, and build-to-learn versus build-to-earn. Two other sources listed against 1.6 in earlier planning were reviewed and routed out: the NLM-lessons doc (technical, not psychology) and the board session notes (org and stakeholder dynamics).

### Key discussion

The frame: habit is the goal, trust is the engine. We want System 3 to become the researcher's default first stop and bookmarked home. The mechanism that forms that habit is trust: every question returns a verifiably correct, cited answer faster than navigating five databases by hand. This reconciles the Hook model and the adoption-gap doc rather than choosing between them, and it grounds the Hook variable reward in the citations-non-negotiable rule and the Phase 2 moat test.

Metric: default first stop, measured by return-rate-per-question-occasion, not daily-active use. Nir's own bar is weekly, and researchers ask episodically, so a daily-active metric would punish the system for a cadence it does not control.

Tension resolved: lowest-friction input (a plain-language question) versus "not a chatbot." These reconcile because "not a chatbot" is carried by the answer format (a structured, provenance-forward research brief) and the surrounding surface (a workspace home with saved queries and suggested competency questions), not by the input box, which stays chat-simple.

Investment loop: chosen full in v1 (saved queries plus feedback-driven personalization). This revises the 2026-05-07 SOUL.md decision, which had deferred learned personalization (USER.md and MEMORY.md) to v1.1. The capability is now v1 scope; the file-versus-database mechanism is a Phase 4 tech-spec detail. Flagged as the heaviest v1 item and the primary descope candidate if the build runs long.

### Decisions made (all logged to DECISIONS.md, 2026-07-21)

1. Adoption frame: habit is the goal, trust is the engine.
2. Adoption metric: default first stop, not daily-active.
3. Experience model: research assistant, not a chatbot.
4. Investment loop full in v1, revising the SOUL.md personalization deferral.
5. Source routing: NLM lessons to the tool and architecture set, board notes to Phase 2.
6. Differentiation anchor: data plus provenance is the moat; personalization compounds it because it is fused with the data, and is not the standalone differentiator.

## Step 1.7: contractor documents (7 sources)

### Agenda

1. Set the lens for the whole contractor package (Track 1 versus Track 2)
2. Tag the NFR baseline (10 categories) for the POC
3. Reconcile the NLQ approach with our earlier query-classification decisions
4. Set federation scope for v1 and confirm NL from day one
5. Decide which of Anne's evaluation outcomes become requirements

### What we reviewed

Seven contractor sources: the NFR baseline (10 categories), the NLQ approach (6 ranked options), the April 21 KG-Tech WG meeting notes (D1 through D5), the July 07 K3 US1/US2 review package (17 competency questions, a per-question data model, still at design-review), and Anne's evaluation playbook (the June 24 Lunch and Learn).

### Key discussion

The lens. Every contractor document describes the official NLM track (Track 2): an RDF plus SPARQL system, GraphQL as the public surface, four federated modular graphs, scoped to glucose metabolism and GLP-1, still at design-review with no running graph. System 3 is different: an already-built property graph (AGE, openCypher, whole NCBI corpus) with a search agent and UI on top. So we mine the package for patterns, NFRs, and evaluation criteria, but do not inherit its architecture. Extends the existing infrastructure-template-not-blueprint decision to the whole package.

NFR baseline. Filtered the 10 categories through three buckets instead of the contractor's own POC/MVP tags: A out of scope (graph build and curation, which is System 1/2), B POC must-have (the query, answer, guardrail, and provenance surface), C defer past POC. Security splits: trust-guardrails (read-only enforcement, provenance integrity, cost caps, prompt-injection defense) stay in the POC because trust is the product; enterprise IAM (roles, session expiry, access review) defers; basic auth returns for v1 because the investment loop needs accounts.

NLQ approach. Reconciled the contractor's "build toward rank 1 typed IR, operate like rank 2 CQ templates" with our earlier decisions (simplified classification, not full IR; a dedicated LLM call generates Cypher). Landed on a confidence-layered hybrid: the POC leads with schema-aware Cypher generation plus a validator (proves the plain-language differentiator, harvests real questions), and verified templates are layered on for the roughly ten tier-1 must-pass questions while hardening to v1 (determinism where we are graded). Generated queries that prove correct get promoted into the template library, which is the feedback loop. Also locked a backend-agnostic principle: the agent produces a database-neutral plan and the query language lives only inside the tool, so swapping AGE for ArangoDB or another store is a contained change.

Federation and NL. v1 federation scope is exactly the three data layers (Layer 1 graph, Layer 2 NCBI APIs, Layer 3 enrichment); external non-NCBI knowledge-graph federation defers to v2. NL from day one confirmed.

Anne's evaluation playbook. Adopted the milestone ladder as the definition of success, at two altitudes: the framework (four climbing gates, each serving a named stakeholder) is a hard PRD requirement in the success-metrics and acceptance-criteria sections; the operational numbers (thresholds, per-question mapping, scoring) live in the Phase 2 evaluation playbook so the locked PRD does not reopen to tune a metric. Evaluation is a process, so PRD success metrics are stakeholder-segmented.

### Decisions made (all logged to DECISIONS.md, 2026-07-21)

1. Lens: the contractor package is Track 2, mined for patterns and criteria, not inherited as architecture.
2. NFR baseline: three-bucket filter; bucket B is the query-and-answer surface.
3. Security posture: trust-guardrails in the POC, enterprise IAM deferred, basic auth for v1.
4. Backend-agnostic design principle: query language sealed inside the tool.
5. Query-generation strategy: generation-first hybrid, verified templates for the tier-1 must-pass questions.
6. Federation scope: the three data layers for v1; external KG federation to v2.
7. Evaluation adoption: the milestone ladder at two altitudes, stakeholder-segmented success.

## Step 1.13: LLM legal and compliance obligations

### What we decided

The last Phase 1 step. Legal obligations are a separate gate that binds the federal production path, not a personal Track 1 prototype on public data, so 1.13 is a two-lane split.

- Track 1 (now): permissive licenses default (use-restricted allowed with tracking, research-only barred, Llama capped-commercial fine); no PHI so no BAA for v1; zero-retention inference terms as a should-have; OpenRouter is a Track 1 hosting choice; model choice stays a config swap.
- Country-of-origin (Option A, Monideep decided): Track 1 uses the strongest models now including Chinese-origin (DeepSeek, Kimi, GLM, Qwen), with compliant US or allied alternatives (Devstral 2, Nemotron 3, Gemma 4, Llama) benched in parallel and a config-swap migration path.
- Production / Track 2 (deferred, designed toward): US or allied-origin only; FedRAMP, FISMA, ATO, OMB memos; BAA if PHI, SOC 2, residency.
