## Decisions (all logged in DECISIONS.md, 2026-07-22)

The moat test is the primary selection and tiering bar, above persona coverage and raw usage frequency.

It operates as a structured bar, not a flat score: provenance and deterministic are pass/fail gates; the one design-time ranking dimension is cannot-just-Google, scored by an external test (run the question against a general tool; if it cannot produce the cited answer, it passes); learn-from-the-system and loop-human-behavior defer to the Step 2.5 online loop; cross-database span was rejected as us-imposed bias. Deterministic means reproducible and verifiable given the data state at query time.

The PoC moat set is a deliberately small, cross-database-biased subset that proves the wedge, separate from the full set of queries the system handles.

Assemble-not-classify: moat CQs surface cited evidence and never render a clinical verdict (the Phase 1 forbidden-outputs boundary at the CQ level). Q1 and Q2 reworded to evidence-assembly.

No compute tools in v1: the three execution-heavy questions (Q2 VCF, Q7 SRA similarity, Q9 BLAST) defer to a fast-follow set, added later with pinned fixtures.

v1 must-pass moat set locked at seven: Q1 (reworded), Q3, Q4, Q5, Q6, Q8, Q10. Covers all three wedge types and personas 1, 2, 3, 4, 6, 8, 10, 11. Q1 carries a coordinate-range feasibility flag; drops to six if infeasible.

Two supporting decisions: no local MCP servers for inbound data (direct Python tools; MCP is an outbound delivery format only), and a full current-state deep dive of the NCBI, Datasets, and enrichment APIs added to Phase 4 as Step 4.0.
