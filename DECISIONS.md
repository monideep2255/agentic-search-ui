# Decisions

Architecture and implementation decisions for System 3 (search agent, API, UI). Append-only table, most recent at the bottom.

This repo was created from `agentic-search-data-engineering` (System 1+2) as a structural template. System 1+2 decision history lives in that repo.

## Table

|------|----------|------------------------|-----|

Read-only enforced at connection level (kg_reader role). |

Sub-query decomposition (LangGraph subgraph, not a separate agent) is the planned upgrade for deep research queries needing 10+ sequential tool calls. | 2026-05-07 | Result compression in the harness before re-injecting tool results into agent context | Pass full tool results into context | A Cypher query returning 500 rows would consume the context window and cause hallucination of remaining data. Already specified in system-design-patterns rule (output truncation for large results). |

At 10 rps, caching was survival-critical and concurrent users competed for budget. At 100 rps, 10 concurrent users still get 10 calls/sec each. Fan-out across 5 databases in parallel is comfortable. Planner budget enforcement still matters for cost/latency, but throttling is no longer the binding constraint. Variation Services API remains at 1 rps (IP-based, separate limit). |
| 2026-05-07 | Two-API strategy for Layer 2: E-utilities (ClinVar, PubMed, OMIM) + Datasets API v2 (Gene, Genome, Orthologs, Taxonomy) + Variation Services (dbSNP) | E-utilities only; Datasets API only; single unified client | E-utilities and Datasets API cover complementary domains with no overlap. | 2026-05-07 | PubTator3 REST API replaces all local NER/normalization (GNorm2, tmVar3, AIONER) | Build local NER stack (65GB+ memory, mixed Java/Python) | PubTator3 wraps GNorm2, tmVar3, AIONER, and BioREx behind one REST endpoint. | 2026-05-07 | NCBI KG reference repo used as infrastructure template (UI, observability, guardrails, testing), not as architecture blueprint | Copy the NL-to-Cypher pipeline architecture; build everything from scratch | The repo's 7-stage NL-to-Cypher pipeline is a monolithic 2-day prototype, not suitable as architecture. | 2026-05-07 | Simplified query classification (query class + target entities + tool list) from Plan step instead of contractor's full typed query-plan IR | Full typed IR with cqClass, focusEntities, joins, constraints, modules, resultShape, provenanceRequired; no classification at all | The contractor's typed IR is a comprehensive JSON contract between language understanding and execution. | 2026-05-07 | Schema slicing for Plan step: send only relevant graph schema portions to the LLM, not the full schema | Send full schema every time; hardcode schema in prompts | AGE graph has 14 edge types and dozens of node properties. Hand-authored for V1, formalized into a builder when query complexity demands it. Concept adopted from contractor's context-pack builder but implemented as a simple function, not the full builder. |
| 2026-05-07 | Contractor's 8-layer architecture not adopted; 5-step agent loop provides equivalent coverage | Adopt full 8-layer architecture (Entry, Guarded Planning, Context Retrieval, Public Surface, Backend Compilation, Knowledge Modules, Provenance Response, Governance) | The 8-layer architecture was designed for an enterprise team with federation, multiple knowledge modules, and a GraphQL-only public surface. Our 5-step loop (Guard, Think, Plan, Act, Write) covers the same concerns with less ceremony. Guard = Entry + governance. Think + Plan = Guarded Planning + Context Retrieval. Act = Backend Compilation + Knowledge Modules. Write = Provenance Response. Fewer layers means fewer abstraction boundaries to maintain in a solo build. |
| 2026-05-07 | NL-to-Cypher separation principle adopted: Plan step produces structured output, tools compile to Cypher; LLM never generates Cypher directly | LLM generates raw Cypher from natural language (reference repo pattern); full typed IR intermediary (contractor pattern) | Key learning from contractor architecture: natural language should never directly produce database queries. Intermediate representation prevents prompt injection into queries, enables validation before execution, and lets tools apply domain-specific rewrites (GO-term expansion, edge label enforcement). In our architecture: Plan step reasons about what to query (structured output), cypher_query tool handles how to query it (Cypher compilation with guardrails). |
| 2026-05-07 | V1 shoring-up recommendations (6 items) are not System 3 blockers; all are data pipeline fixes for the next graph reload | Treat items 1-3 as blockers that must be fixed before System 3 development | The graph is already loaded and running (115M nodes, 693M edges). | 2026-05-07 | Layer 2 serves as authoritative fallback when Layer 1 graph data is suspect; agent validates graph results before synthesis | Trust Layer 1 data blindly; fail and show error on bad data | Layer 1 is a periodic snapshot that can have stale data, corrupted fields (MedGen names showing "SNOMEDCT_US"), wrong categories (NamedThing stubs), or missing nodes. Layer 2 (live NCBI APIs) is always current and authoritative. The agent's Write step validates graph results and falls back to Layer 2 for correction when data looks suspect. Same pattern handles known issues (MedGen names, NamedThing stubs) and future unknown data quality issues. Principle: Layer 1 for speed, Layer 2 for correction, Layer 3 for enrichment. |
| 2026-05-07 | Graceful degradation: the user must never see nothing; always synthesize from whatever layers responded | Show error message on failure; show nothing and let user retry | If Layer 1 has bad data, fall back to Layer 2/3. | 2026-05-07 | LLM is narrative controller only; harness assembles citations and provenance deterministically from structured tool outputs | LLM handles both narrative and citations; fully template-based output with no LLM | The LLM writes natural language narrative with placeholder markers ([1], [2]). The harness (code) maps markers to verified sources from tool results (source_url from graph nodes, NCBI API record URLs, PubTator section refs). If a marker doesn't match a real tool result, the harness strips it. The user never sees a citation the harness can't verify. Balance: LLM has narrative flexibility (not rigid templates), but citations are always deterministic and correct. This also reduces cost: if synth tier only writes narrative from pre-assembled data, it can use a cheaper model. Determinism = financial viability. |
| 2026-05-07 | Tools are direct Python functions inside FastAPI for V1; MCP delivery format wraps the same functions behind MCP protocol separately | Tools ARE MCP servers from the start; raw API calls without tool abstraction | Tools as direct functions avoids inter-process communication overhead and keeps V1 simple. | 2026-05-07 | Verification loop between Write step and user: rules-based checks on every response, LLM-as-judge reserved for eval harness only | No verification (trust LLM output); LLM-as-judge on every response | After Write step produces structured JSON output but before user sees it, deterministic checks run: every citation marker maps to a real tool result, CURIEs resolve, source_urls are valid, answer addresses query intent. | 2026-05-07 | Cypher generation uses a dedicated LLM call inside the cypher_query tool, not the main agent loop; 3-stage internal pipeline (receive intent, generate Cypher, validate then execute) | LLM generates Cypher as free-form output from the main agent; fully template-based Cypher with no LLM | The NL-to-Cypher separation principle (Plan step decides WHAT, tool decides HOW) requires the tool to translate structured intent into actual Cypher. Generated Cypher is validated (syntax check, forbidden keywords, edge labels required, row limit) before execution. If validation fails, one retry with the error message, then fail gracefully. The main agent never sees or generates Cypher. |
