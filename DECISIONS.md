# Decisions

Architecture and implementation decisions for System 3 (search agent, API, UI). Append-only table, most recent at the bottom.

This repo was created from `agentic-search-data-engineering` (System 1+2) as a structural template. System 1+2 decision history lives in that repo.

## Table

| Date | Decision | Alternatives considered | Why |
|------|----------|------------------------|-----|
| 2026-05-05 | PostgreSQL + Apache AGE knowledge graph accessed read-only via psycopg2; no graph write operations in this repo | Embed graph queries directly in the agent; use a GraphQL layer over AGE; use Neo4j client | AGE graph already exists on Hetzner CPX42 (115M nodes, 693M edges). psycopg2 is the simplest client with no additional infrastructure. Read-only enforced at connection level (kg_reader role). |
| 2026-05-05 | Multi-model harness with 3 tiers (guard/plan/synth) via LiteLLM | Single model for everything; two tiers (cheap/expensive); direct Anthropic/OpenAI SDK calls | Guard tier keeps input validation cheap (~$0.001/query). Plan tier balances cost/capability for tool selection. Synth tier uses the strongest model only for final answer generation. LiteLLM abstracts provider differences. Hard cost caps at every tier prevent runaway spend. |
| 2026-05-05 | LangGraph for agent orchestration (not raw LangChain or custom state machine) | LangChain AgentExecutor; custom async state machine; CrewAI; AutoGen | LangGraph gives typed state, conditional edges, and built-in persistence without the abstraction overhead of AgentExecutor. Supports streaming natively. The 5-step agent loop maps directly to a LangGraph graph with conditional routing. |
| 2026-05-05 | FastAPI + SSE streaming (not WebSocket) for chat responses | WebSocket; HTTP polling; gRPC streaming | SSE is simpler than WebSocket for unidirectional server-to-client streaming. Works through CDNs and proxies without upgrade negotiation. sse-starlette integrates with FastAPI's async generators. Client reconnection is built into the EventSource spec. |
| 2026-05-05 | Data source adapter pattern: each source implements only applicable adapters (Query, Facet, Citation, Relationship, Streaming) | Single monolithic interface with NotImplementedError; separate class per operation; duck typing | Adapters are capability declarations. The agent checks adapter availability before attempting operations. No "not implemented" exceptions, no silent no-ops. Each source is honest about what it can do. |
| 2026-05-05 | Production-level SDLC standards from day 1 via 6-lens dev-standards skill (adapted from NWS enterprise standards) | Add standards later when closer to production; use a lighter checklist; no formal review process | Technical debt compounds exponentially. Patterns established in week 1 become the baseline forever. The 6-lens approach (security, testing, quality, PR readiness, deployment, hardening) catches issues before they become architectural problems. Cost of applying standards early is near zero; cost of retrofitting is high. |
| 2026-05-05 | Cypher queries must always specify edge labels explicitly (never untyped `[r]`) | Allow untyped edges and rely on planner pruning; force all queries through a builder abstraction | AGE compiles untyped edges to UNION ALL over all 14 edge label tables (693M rows). Specifying the edge label lets the planner touch only the relevant table. This convention alone takes queries from minutes to milliseconds. Learned from Gate 3 indexing work in System 1+2. |
| 2026-05-07 | Innovation proposal is the starting ground for V1: all scope elements and requirements apply | Treat the proposal as aspirational and cherry-pick for V1 | The innovation proposal defines the minimum viable product. Even though V1 is personally funded, it must demonstrate the full system to serve as evidence for the NCBI track. |
| 2026-05-07 | All 5 delivery formats in scope for V1: web UI, GraphQL API, MCP server, KGX export (System 1/2 graph only), CLI agent | Web UI only for V1, add others later | The proposal specifies 5 formats. KGX export is scoped to the existing Hetzner graph (not a full Neo4j export). All formats are part of demonstrating the system works end to end. |
| 2026-05-07 | FastAPI over Django for V1 backend | Django (NCBI standard stack) | Agent loop requires native async for concurrent API calls, LLM streaming, and LangGraph integration. Django's async support is bolted on, and its ORM adds no value (we use raw Cypher, not SQL models). Core agent code is framework-agnostic, so porting to Django for NCBI Track 2 is bounded work (HTTP layer only, not the agent logic). |
| 2026-05-07 | Hybrid API surface: REST + SSE for chat streaming, GraphQL (via Strawberry) for structured programmatic access | GraphQL only; REST only | Chat is a conversation (stateful, streaming) and fits REST+SSE naturally. Programmatic access to nested biomedical data (genes, variants, diseases, papers) fits GraphQL. Both served from the same FastAPI app, sharing auth middleware and underlying tools. |
| 2026-05-07 | NCBI strategic directives (FY26 guiding principles, Gold Standard Science, AI Action Plan) are real requirements for V1, not aspirational alignment | Treat as nice-to-have positioning | These define how the system must behave: simplify discovery (FY26), reproducibility and transparency (Gold Standard), AI-ready datasets (AI Action Plan). V1 is personally funded but must meet these standards to serve as evidence for the NCBI track. |
| 2026-05-07 | Open-source LLMs preferred for the running system; commercial models reserved for development debugging only | Commercial models (Anthropic, OpenAI) for all tiers; single model for everything | Harness engineering philosophy: a strong harness (guardrails, structured tool calls, citation assembly) reduces the capability required from each model. Guard tier is classification (tiny model). Plan tier is structured output (mid-range). Synth tier assembles pre-fetched cited data, not open-ended reasoning. Open-source keeps the running system cost at single digits per month. |
| 2026-05-07 | OpenRouter as the inference provider, accessed via LiteLLM in application code | Groq free tier; direct provider API keys (Groq + Together + Fireworks); OpenRouter without LiteLLM; LiteLLM with direct provider keys | OpenRouter provides a unified API gateway to 100+ models (one API key, one billing). LiteLLM stays as the in-code SDK abstraction (typed calls, cost tracking, retry logic, LangGraph integration). Each layer does what it's good at: LiteLLM controls which model gets called per tier, OpenRouter handles provider routing and fallbacks. Avoids managing multiple provider accounts. Pricing for open-source models is comparable to direct access. |
| 2026-05-07 | V1 budget target: ~$100 total for 2-month build (not counting Claude Code Max subscription) | Original $300-500 estimate | Hetzner CPX42 ~$60 (already running). Railway free (1-year plan). LangSmith free tier. LLM inference via OpenRouter on open-source models: single digits per month. Claude Code Max ($100/mo) is a personal development tool, not a project cost. |
| 2026-05-07 | Single orchestrator agent with 3-tier model routing for V1; sub-query decomposition as the upgrade path for complex queries | Multiple independent agents; orchestrator + domain worker agents; single model for everything | V1 competency questions need 3-5 tool calls per query. One agent handles this trivially. Multi-agent adds inter-agent communication, handoff protocol, and error handling across agent boundaries for no benefit at this scale. The harness provides multi-model routing (different model per step) without multi-agent complexity. Sub-query decomposition (LangGraph subgraph, not a separate agent) is the planned upgrade for deep research queries needing 10+ sequential tool calls. Trigger: >20% failure rate on deep research class queries in the golden dataset. |
| 2026-05-07 | Model selection deferred to build phase (Phase 6); harness pattern locked now, specific models selected via golden dataset ablation | Lock specific models now | The harness pattern (3 tiers, LiteLLM + OpenRouter) is the decision. Specific models (DeepSeek R1, Kimi K2, Gemma 4, Llama 4, Qwen3) are candidates to be benchmarked against the golden dataset during build. Open-source Chinese models (DeepSeek, Kimi K2, Qwen) are competitive on structured tasks and preferred for cost. Locking models before measuring performance wastes the harness's main advantage: swapping models is a config change. |
| 2026-05-07 | Streaming with typed SSE events (status, tool_result, token, citation, done) and hard latency budgets per query class | Non-streaming request/response; WebSocket streaming | Users must see agent progress from T+0.1s, never a blank screen. SSE event types give the UI structured data to render progress (status updates, citation chips, streaming tokens). Hard latency budgets enforced by the harness: lookup 5s, single-hop 10s, multi-hop 30s, deep research 2min. If exceeded, synthesize with partial results and explain what timed out. Already decided SSE over WebSocket in Step 1.1; this decision adds the event type schema and latency enforcement. |
| 2026-05-07 | Three-layer data architecture confirmed: Layer 1 (AGE graph, <10ms), Layer 2 (NCBI APIs, 100-500ms), Layer 3 (enrichment APIs, 200ms-2s) | Ingest all databases into graph; API-only with no graph; two layers | The architecture doc and three-layer doc agree. 5 databases in Layer 1 (Gene, PubMed, ClinVar, MedGen, Taxonomy) provide the graph skeleton for fast traversals. Layer 2 (dbSNP, Protein, PMC, OMIM, GTR, GEO, dbVar, Assembly) are leaf nodes reached via live API at query time. Layer 3 (PubTator3, LitVar2, LitSense, ClinicalTrials.gov) augments answers with enrichment. This is already built and running on Hetzner. |
| 2026-05-07 | Parallel tool execution via asyncio.gather for independent API calls | Sequential tool calls | Three API calls at 300ms each: 300ms total (parallel) vs 900ms (sequential). Independence between calls is determined by the planner output. The LangGraph agent loop dispatches independent tool calls in parallel by default. |
| 2026-05-07 | Result compression in the harness before re-injecting tool results into agent context | Pass full tool results into context | A Cypher query returning 500 rows would consume the context window and cause hallucination of remaining data. The harness summarizes results (e.g. "14 genes found: HNF1A, HNF4A, GCK... Top pathogenic variants: 47 across 8 genes") before the agent sees them. Keeps context small regardless of result volume. Already specified in system-design-patterns rule (output truncation for large results). |
| 2026-05-07 | NCBI API rate limit upgraded from 10 to 100 req/sec (admin access); architecture unchanged but headroom significantly relaxed | Keep at 10 rps and design around it | Admin access on the API key enables 100 rps. At 10 rps, caching was survival-critical and concurrent users competed for budget. At 100 rps, 10 concurrent users still get 10 calls/sec each. Fan-out across 5 databases in parallel is comfortable. Planner budget enforcement still matters for cost/latency, but throttling is no longer the binding constraint. Variation Services API remains at 1 rps (IP-based, separate limit). |
| 2026-05-07 | Two-API strategy for Layer 2: E-utilities (ClinVar, PubMed, OMIM) + Datasets API v2 (Gene, Genome, Orthologs, Taxonomy) + Variation Services (dbSNP) | E-utilities only; Datasets API only; single unified client | E-utilities and Datasets API cover complementary domains with no overlap. Gene records are richer via Datasets API (GO terms, transcripts). ClinVar, PubMed, OMIM are only available via E-utilities. dbSNP individual lookups use the Variation Services API (1 rps, separate from E-utilities). Three APIs, three rate limit pools, three tools. |
| 2026-05-07 | PubTator3 REST API replaces all local NER/normalization (GNorm2, tmVar3, AIONER) | Build local NER stack (65GB+ memory, mixed Java/Python) | PubTator3 wraps GNorm2, tmVar3, AIONER, and BioREx behind one REST endpoint. Accepts raw text, returns normalized entities with database IDs. Free, no auth. Avoids 60GB JVM heap for GNorm2 alone. Already confirmed in NCBI repos deep dive. |
