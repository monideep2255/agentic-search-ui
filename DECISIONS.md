| 2026-04-20 | Phase 4.0 rsync wrapped in `scripts/rsync-retry.sh` auto-retry loop with `--partial --inplace` flags, plus `--append` flag after the attempt-24 hang. | Single rsync invocation with manual rerun on drop; chunk the 144 GB file into ~150 separate 1 GB files and rsync each as its own session; use `bbcp` or `aspera` for WAN-optimized transfer; switch to scp + manual restart (no resume); use cloud storage as intermediary (S3/R2/Hetzner Storage Box upload then VPS download) | Single invocation breaks because every SSH session dies at ~1-1.4 GB on home Wi-Fi (likely consumer router NAT table eviction at ~5 min wall-clock per TCP connection). Manual rerun would require ~100 reconnections by hand. The retry loop with `--partial --inplace` survives drops and resumes from the last committed byte automatically; tested across attempts 1-23 at sustained ~14 MB/s. The `--append` flag was added after attempt 24 hung silently for 33 min (root cause not isolated; possibly a stuck SSH handshake invisible to `--timeout=600`); skipping rsync's default delta-checksum re-verification on resume removes 30-90 sec of local-rescan overhead per attempt. Trade-off: `--append` trusts the partial bytes are valid (no checksum verification of overlap); acceptable because `--inplace` guarantees direct disk writes with no temp-file shenanigans. The retry loop has a 200-attempt cap to prevent runaway in case of unrecoverable failure. Without these mitigations, the transfer would be unfinishable in any reasonable wall-clock window. Rationale and full forensics in `docs/learnings.md` Phase 4.0 execution section + Understanding subsection on the retry pattern. |
| 2026-04-20 | When rsync attempt hangs silently for >10 min during Phase 4.0 transfer, kill the specific stuck rsync.exe + ssh.exe processes (not the bash retry loop).
