# Decisions

Architecture and implementation decisions made during this project.

| Date | Decision | Alternatives considered | Why |
|------|----------|------------------------|-----|
| 2026-04-06 | PostgreSQL + Apache AGE for graph storage (supersedes Neo4j decision below) | Neo4j Community, Neo4j AuraDB | Neo4j Community needs 64GB+ RAM for large graphs. PostgreSQL + AGE is disk-based and handles 150M nodes on 8GB RAM. Saves $100-200/month on hosting. Cypher syntax still supported via openCypher in AGE. |
| 2026-04-06 | Neo4j Community Edition for local dev (SUPERSEDED 2026-04-06 by PG+AGE above) | AuraDB (cloud), PostgreSQL + AGE | Free, local, full Cypher support. Switch to AuraDB or self-hosted for production. PostgreSQL + AGE is the budget option if Neo4j costs are prohibitive. |
| 2026-04-06 | Start with 1-month scope: Gene + ClinVar + MedGen only | Full 6-database scope from week 1 | Ship something real in 4 weeks. PubMed download alone takes 4-8 hours. Add remaining databases in month 2. |
| 2026-04-06 | KGX format for pipeline output (nodes.tsv + edges.tsv) | Direct Neo4j Cypher INSERT, JSON-LD | KGX is the standard for BioLink-compliant graph exchange. System 1 and System 2 stay decoupled. |
| 2026-04-06 | LinkML for schema definition | Raw BioLink YAML, Pydantic models | LinkML generates validators automatically. BioLink uses LinkML natively. Schema-as-code. |
| 2026-04-06 | SNP: clinical subset only (~500K) via ClinVar cross-references | Full dbSNP (1.2B records) | 1.2B records are too large for a 2-month alpha. Clinical subset covers the queries that matter. Expand later. |
| 2026-04-06 | PubMed: 2024-2026 subset first | Full 40M article baseline | Full baseline download is 30GB compressed, 4-8 hours. Start small, prove the pipeline, then expand. |
| 2026-04-06 | Apache 2.0 license | MIT, GPL | Permissive (commercial use allowed), has patent protection, NCBI can adopt without legal friction. |
| 2026-04-06 | This repo holds System 1 + System 2 only; System 3 in a separate repo | Single monorepo for all three systems | System 3 (search agent, FastAPI, LangGraph, UI) has different deployment cadence and dependencies. Decoupling lets data pipelines and graph load run independently of agent layer. |
| 2026-04-06 | Adopt 9-step pipeline pattern from canonical reference (`reference/ncbi_ai_agents-ncbi-kg/KG/pipeline/`) over reinventing | Custom 5-step pattern from scratch | Reference pipeline already produces a BioLink 4.x compliant KG with 82K nodes / 263K edges. Idempotent download, dedup, dangling-edge validation, MONDO stub injection, KGX export are all working. Copy patterns directly. |
| 2026-04-06 | BioLink 4.x: 8 node categories, 15 edge predicates (verbatim from reference) | Define our own subset, use BioLink 5.x | Reference pipeline schema is validated against real glucose metabolism KG. Categories: Gene, Protein, SequenceVariant, Disease, BiologicalProcess, MolecularActivity, CellularComponent, Pathway. We extend rather than replace. |
| 2026-04-06 | DataFrame-based assembly with pandas (per reference pipeline) | Direct streaming to graph, dict-of-dicts | Reference uses pandas DataFrames for nodes/edges, dedups by id and (subject, predicate, object), validates dangling edges. Mature, tested, debuggable. |
| 2026-04-06 | Canonical CURIE prefixes: NCBIGene, NCBIProtein, ClinVar, MONDO, UMLS, GO, Reactome, MeSH | Custom internal IDs | Match reference pipeline conventions so KGX files merge cleanly with the existing graph and downstream consumers. |
| 2026-04-13 | Full PubMed baseline, 40M articles (SUPERSEDES 2026-04-06 subset decision) | 2024-2026 subset (~5M articles) | Subsetting creates millions of dangling gene2pubmed edges pointing to nonexistent article nodes. Full baseline avoids this. Download overnight. |
| 2026-04-13 | Full dbSNP, all 1.2B records (SUPERSEDES 2026-04-06 clinical subset decision) | Clinical subset only (~500K via ClinVar cross-refs) | Population frequencies, functional annotations, and variant type data are needed for SME queries. Full dbSNP is the only way to answer allele frequency questions. |
| 2026-04-13 | AGE hardware estimate updated to 1.4B nodes on 8-16GB RAM (SUPERSEDES 150M figure in 2026-04-06 row) | Original 150M estimate | Full dbSNP adds 1.2B variant nodes. AGE is disk-based so RAM is not the bottleneck. Neo4j now requires 256GB+ RAM ($500+/month), ruling it out entirely at this scale. |
| 2026-04-13 | Schema is a shared artifact in `schema/` at repo root, consumed by both System 1 and System 2 | Schema owned by System 2, System 1 consumes it | System 1 needs the schema before it can build any mapper. Neither system owns it; both depend on it. Defined once as step 1a. |
| 2026-04-13 | Full Taxonomy tree (2.9M organisms) from the start | Phase 1: ~10 hardcoded model organisms, full tree deferred | All data means all data. Taxonomy is only ~500MB compressed. No reason to subset. |
| 2026-04-13 | PubChem, SRA, dbGaP excluded from all layers | Include PubChem in Layer 2 | Per innovation proposal: SRA is raw sequencing reads (analysis pipelines, not search). dbGaP is controlled-access (IRB approval required). PubChem is community-submitted with varying curation (poor fit for authoritative provenance). |
| 2026-04-13 | All data principle: download everything on FTP for each Layer 1 database, no cherry-picking | Download only files needed for known edges | Storage and processing strategy are implementation decisions. Download scope is non-negotiable. If a file exists in the FTP directory, download it. |
| 2026-04-13 | Rename directories: `system-01-data-pipelines/` and `system-02-knowledge-graph/` | `data-pipelines/` and `knowledge-graph/` | Naming convention makes the system boundary explicit. Matches the System 1 / System 2 terminology used throughout the plan and proposal. |
| 2026-04-13 | Branch-per-phase git workflow with MR review before merging to main (SUPERSEDES direct-to-main) | Work directly on main, merge without MR | Bossman mode executes phases autonomously. Phase branches give the user a review gate before any agent-written code reaches main. One MR per phase; branch deleted after merge. |
| 2026-04-14 | ClinVar parser uses tabular variant_summary.txt.gz, not ClinVar XML | ClinVar XML (variant_set_release.xml.gz) | variant_summary.txt.gz is a flat TSV with one row per variant, streamed line by line for memory safety. XML is deeply nested, requires a SAX parser, and is harder to maintain. Tabular file covers all fields needed for BioLink node and edge production. |
| 2026-04-14 | Phases 1.2, 1.3, 1.4 (Gene, ClinVar, MedGen ETL) built on a single combined branch | Three separate phase branches | The three pipelines share the mim2gene_medgen cross-reference file and are tested together via the merge step. Building on a single branch avoids merge conflicts when the shared cross-reference parsers are updated mid-session. |
| 2026-04-14 | MedGen pipeline rewrites CUI references in edges to canonical IDs after MONDO promotion | Keep MedGen:{CUI} as the edge endpoint | parse_id_mappings promotes a MedGen concept to MONDO:{id} when a MONDO mapping exists. Edges built before promotion use the CUI as subject/object. A rewrite pass in pipeline.py updates all edge references to the final canonical ID so the graph is internally consistent. |
| 2026-04-14 | bossman-mode uses single-agent sub-agents for parallel pipeline builds, not full agent teams | Full agent team per pipeline (separate builder + judge per pipeline) | For code generation tasks in Phase 1, a single sub-agent per pipeline is sufficient. Full agent teams (with dedicated judge agents per pipeline) add coordination overhead without quality benefit at this stage. Revisit for Phase 2 if pipeline complexity increases. |
| 2026-04-14 | KGX files are intermediates, not the final product. Graph database (PostgreSQL + AGE) is the end target. | Keep KGX files as permanent archive alongside the graph | KGX files for 6 databases would consume 300-500GB. Disk is limited to 434GB. Once data is loaded into AGE, the KGX files serve no purpose. They can be regenerated from the original FTP cache if needed. |
| 2026-04-14 | Reorder: load 5-database graph into AGE (Phase 4.0) before building dbSNP pipeline (Phase 3.0) (SUPERSEDES original Phase 3 then Phase 4 order) | Build all 6 pipelines first, then load everything into AGE at the end | dbSNP KGX output is 200-400GB, which exceeds available disk if the other 5 databases' KGX files are still present. Loading the first 5 into AGE and deleting their KGX intermediates frees ~50-100GB before dbSNP starts. dbSNP then loads incrementally per chromosome directly into AGE. |
| 2026-04-14 | Gene pipeline streams edges to disk per parser batch instead of accumulating all edges in memory | Collect all edges in a list, then export in one pass | Gene pipeline produces 278M edges. At ~500 bytes per dict, that is ~130GB in memory. The process was OOM-killed during Gate 1. Streaming via append_edges() writes each parser's edges to disk immediately and frees the list. |
| 2026-04-16 | 5-database merge lives in a new `merge/` pipeline directory with its own CLI (`merge-etl`) | Add the orchestrator inside `shared/` next to `merger.py` | The existing `shared/merger.py` exposes primitives (merge_kgx, inject_stubs, validate_merge). The orchestrator composes them and needs a CLI entry point, matching the pattern of gene/, clinvar/, etc. Keeping `shared/` reserved for primitives keeps the boundary clear. |
| 2026-04-16 | Merger CURIE prefix map extended with MeSH, ClinVar, UMLS; stubs infer BioLink category from prefix | Hardcode MeSH-only extension | Phase 2.2 introduces PubMed (Article, OntologyClass) and Taxonomy (OrganismTaxon) into the merged graph. Stub inference must cover every prefix present in the 5 pipelines so dangling refs resolve to correctly-typed stubs. ClinVar and UMLS were missing from the original triangle map and are now included for completeness. |
